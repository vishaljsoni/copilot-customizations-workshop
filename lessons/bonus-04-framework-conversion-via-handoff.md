# Bonus 4 — Framework Conversion via Multi-Agent Handoff

## Goal

Convert the vanilla-JS to-do app to **React + Vite** using a chain of three agents — Plan → Convert → Test — connected by handoffs. This shows how a meaningful, multi-step refactor can be broken into discrete, reviewable stages without writing any orchestration code yourself.

## Why this is interesting

Lesson 2 had two agents and one handoff. Real refactors need more structure: think → do → verify. This bonus chains three custom agents:

1. **`plan` (built-in)** — read-only; produces a numbered conversion plan.
2. **`react-converter` (custom)** — implements the plan; can run `npm` commands; cannot run tests.
3. **`tester` (already exists from lesson 2)** — verifies behavior parity.

Each agent has its own tool restrictions, its own job, and a handoff to the next. Click handoff buttons three times → ship. **You** are the orchestrator; the framework just provides the rails.

> **Caution:** This bonus actually scaffolds a Vite project (`npm create vite`). It's slow on the first run (~30s install). Skip this bonus if you're short on time.

---

## 1. Create the converter agent

Create **`.github/agents/react-converter.agent.md`**:

```markdown
---
description: Convert a vanilla-JS app to React + Vite. Implements an existing plan; does not write tests.
tools: ['edit', 'codebase', 'search', 'usages', 'runCommands']
handoffs:
  - label: Verify with Tester
    agent: tester
    prompt: |
      Update tests for the converted app. The pure state functions in `src/state.js` should still work and have tests. Run `node --test` and confirm they pass.
    send: true
---

# React Converter

You convert vanilla-JS apps to React + Vite. You implement a plan that was generated by a planning agent. You do not write tests; you hand off to `tester` for that.

## How you work
1. Read the most recent message containing a numbered plan.
2. Run `npm create vite@latest react-app -- --template react` (use the integrated terminal).
3. Move pure state from `state.js` into `react-app/src/state.js`. **Do not modify the pure functions** — they should work unchanged.
4. Translate `app.js` into React components in `react-app/src/`:
   - `App.jsx` (top-level)
   - `TodoForm.jsx`
   - `TodoList.jsx`
   - `TodoItem.jsx`
   - `TodoFilters.jsx`
5. Use **functional components + hooks** (no class components).
6. Persist via `useEffect` watching `localStorage`.
7. Translate `styles.css` into `react-app/src/App.css`. Keep BEM naming. Do not introduce styled-components, Emotion, or Tailwind.
8. Confirm the app runs: `cd react-app && npm install && npm run dev`. Stop the server (Ctrl+C). Report the URL it printed.
9. Hand off to `tester`.

## Hard rules
- The converted app must still pass tests against `src/state.js`.
- No additional npm dependencies beyond `react`, `react-dom`, and Vite's defaults.
- No build-step JSX hacks; just standard `.jsx` files.
- If the plan is missing or unclear, do not start. Hand back with a question.
```

Save.

## 2. Use the built-in Plan agent first

The Plan agent ships with VS Code — you don't need to create it. It's read-only.

In the Chat view:
1. New chat session.
2. Agent dropdown → **Plan**.
3. Send:

```
Convert this vanilla-JS to-do app to React + Vite. Keep the pure state functions in state.js unchanged. Preserve all existing functionality (add, complete, delete, persistence, filters, tests). Generate a step-by-step plan with file paths, component names, and the order of operations.
```

The Plan agent produces a numbered plan. **Read it.** This is your last review point before code starts changing.

## 3. Hand off to react-converter

The Plan agent should show a **"Start Implementation"** button (built-in handoff).
- Default target is the `agent` (general) agent.
- We want our custom `react-converter`. Some VS Code versions let you pick the target from the handoff button; in others, edit the pre-filled prompt and add `@react-converter` or manually switch the agent dropdown to React Converter, then submit.

When the React Converter is active, it executes the plan:

- Approve `npm create vite@latest react-app -- --template react` (this scaffolds a new sub-folder; ~30s).
- Approve file edits as it ports state and components.
- Approve `npm install` (this is the only allowed install — it's just Vite's deps).
- Approve `npm run dev` so it can confirm the app boots, then stop.

When done, the agent shows a **"Verify with Tester"** button.

## 4. Tester runs

Click the handoff. The Tester agent runs:
- It checks that `react-app/src/state.js` has the same exports.
- Adapts `state.test.js` paths if needed.
- Runs `node --test`. All previous tests should still pass — the **pure functions never changed**.

## 5. Look at what just happened

```
You wrote: 1 prompt to the planner.
You clicked: 2 handoff buttons.
Result: a working React + Vite app, with tests still passing.
```

You didn't write any orchestration. Each agent had a small, explicit job. You reviewed at three points (plan, conversion result, test result). The framework provided the rails.

## Self-check

- [ ] `react-app/` folder exists with a Vite-scaffolded React project
- [ ] `react-app/src/state.js` is byte-for-byte identical to your old `state.js`
- [ ] The original `state.test.js` still passes against the new state location
- [ ] `cd react-app && npm run dev` boots and the to-do app works in browser
- [ ] You did not write any code; agents did

## Common pitfalls

| Symptom | Fix |
| --- | --- |
| Plan agent produces vague output | Re-prompt with: "Make the plan more specific. List exact file paths and React component names." |
| Converter modifies `state.js` | The plan was wrong. Hand back to Plan, ask for stricter file separation. |
| Tester says imports broken | Test file imports `./state.js` from old location; update to `react-app/src/state.js` or move the test file |
| `npm create vite` hangs | Network issue or interactive prompt; the converter might be waiting for `y/n`. Send "y" to the terminal manually |

## When NOT to use this pattern

Multi-agent handoff is overkill for:
- Renames, typo fixes, single-file edits — use chat directly.
- Anything that doesn't have natural review points (e.g., adjusting CSS).

It shines when there's a clear **plan → implement → verify** structure and you want a checkpoint at each transition.

## Stretch

Add a **fourth** agent — `qa` — that runs Playwright (via MCP) to take screenshots of the original and converted apps and visually diff them. Now your chain is Plan → Convert → Test → Visual QA. Each agent does one job; humans review in between.
